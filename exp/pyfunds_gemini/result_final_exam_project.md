Final Project Assessment Design
===============================

### **Final Project Plan: Real-World Data Analysis with Python**

**Course:** Python Fundamentals
**Project Weight:** 35% of Final Grade (replaces Final Exam)

---

### **1. Project Description**

This project is a culminating assessment designed to replace the traditional final exam. Students will select a real-world dataset from a curated list, formulate three to five specific analytical questions about the data, and write a Python program to answer those questions. The final submission will be a well-documented Google Colab notebook that contains functional code, explains the methodology, presents results clearly, and offers a brief interpretation of the findings.

This project requires students to move beyond simple implementation to genuine analysis, applying their programming skills to solve a tangible problem and communicate their results effectively.

### **2. Objectives**

Upon successful completion of this project, students will be able to:
*   Apply fundamental Python concepts, including data types, control flow, functions, and file I/O, to solve a complex problem.
*   Utilize standard library modules (e.g., `csv`) to process and analyze data from real-world sources.
*   Develop and implement structured, algorithmic solutions to answer specific analytical questions.
*   Write clean, readable, and well-documented code that adheres to professional standards.
*   Analyze a dataset to extract meaningful information and present findings in a clear, organized format.
*   Effectively communicate their methodology, code, and results within a computational notebook.

### **3. Timeline and Milestones**

The project is structured across several weeks to provide scaffolding, feedback, and adequate time for completion.

| Week | Milestone | Weight | Due Date |
| :--- | :--- | :--- | :--- |
| **Week 12** | **Project Launch** | - | N/A |
| **Week 13** | **Milestone 1: Project Proposal** | 5% | End of Week 13 |
| **Week 14** | **Milestone 2: Code Draft Submission** | 5% | End of Week 14 |
| **Week 15** | **Peer Reviews Completed** | 5% | Middle of Week 15 |
| **Finals Week** | **Milestone 3: Final Submission** | 20% | Day Before Final Exam Period Ends |

**Milestone 1: Project Proposal (5%)**
*   **Task:** Submit a 1-2 page proposal outlining your plan. This serves as a checkpoint to ensure your project is on a viable track.
*   **Contents:**
    1.  Your chosen dataset.
    2.  The 3-5 specific, answerable questions you intend to investigate.
    3.  A high-level plan (pseudocode or a bulleted list of steps) for how you will answer *one* of your questions.
*   **Submission Format:** `.pdf` file submitted via Canvas.

**Milestone 2: Code Draft & Peer Review (10%)**
*   **Task:** Submit a draft of your Google Colab notebook containing functional code that attempts to answer your proposed questions. This milestone is graded on the submission of your draft (5%) and the quality of the two peer reviews you provide (5%).
*   **Submission Format:** `.ipynb` file submitted via a Canvas Assignment configured for Peer Review.
*   **Peer Review:** You will be anonymously assigned two peer submissions to review using a guided rubric. The goal is to provide constructive feedback on code readability, structure, and functionality.

**Milestone 3: Final Submission (20%)**
*   **Task:** Submit your final, polished Google Colab notebook, incorporating feedback from the proposal and peer review stages.
*   **Contents:** The notebook must be a self-contained report including:
    1.  **Introduction:** A Markdown cell stating the chosen dataset and the questions being investigated.
    2.  **Code Implementation:** Well-commented Python code cells, logically organized into functions.
    3.  **Results & Analysis:** Markdown cells that clearly present the answers produced by the code and include a brief (1-2 sentence) interpretation for each result.

| Milestone | Deliverable(s) |

### **5. Grading Rubric**

#### **Peer Review Rubric (for Milestone 2)**

| Criteria | Proficient (2.5 pts) | Developing (1.5 pts) | Needs Improvement (0.5 pts) |
| :--- | :--- | :--- | :--- |
| **Clarity & Readability** | Feedback constructively addresses variable names, comments, and overall code structure. | Feedback mentions readability but is vague. | Feedback is unhelpful or does not address readability. |
| **Functionality Check** | Reviewer confirms whether the code runs and correctly identifies any errors encountered. | Reviewer notes that the code runs/doesn't run but provides no detail. | Feedback does not mention whether the code was tested. |
| **Constructive Suggestion**| Reviewer provides at least one specific, actionable suggestion for improvement. | Suggestion is generic (e.g., "add more comments"). | No suggestion is provided. |

#### **Final Submission Rubric (for Milestone 3)**

| Criteria | Excellent (45-50 pts) | Proficient (35-44 pts) | Developing (25-34 pts) | Needs Improvement (<25 pts) | **Weight** |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **1. Functional Correctness**| Code runs without errors, correctly implements the logic, and produces accurate answers to all proposed questions. | Code runs and produces mostly correct answers, with minor logical flaws in one area. | Code runs but has significant logical errors, leading to incorrect answers for multiple questions. | Code fails to run or produces grossly incorrect results. | **50%** |
| **2. Code Quality & Style** | Code is exceptionally clean, well-commented, and adheres to PEP 8. Logic is effectively decomposed into reusable functions. | Code is readable with adequate comments and functions. Minor style inconsistencies may be present. | Code is difficult to read due to poor naming, lack of comments, or monolithic structure. | Code is messy, uncommented, and shows no attempt at organization. | **30%** |
| **3. Analysis & Communication**| The notebook is a polished report. Questions are clear, results are presented professionally, and interpretations are thoughtful. | The notebook presents the results clearly. Interpretations are present but may be superficial. | The notebook contains code and output but lacks clear explanations or interpretation. | The submission is just a code file with no explanatory text. | **20%** |

### **6. Submission Formats and Environment**

*   **Work Format:** This is an **individual** project. All submitted work must be your own.
*   **Development Environment:** **Google Colab**. All development must be done in a `.ipynb` notebook.
*   **Submission Portals:** All deliverables will be submitted through **Canvas**.
*   **Dataset Options:** You must choose one of the curated `.csv` datasets provided on Canvas. Examples include:
    1.  NYC Central Park Squirrel Census
    2.  Top 1000 IMDB Movies (Simplified)
    3.  Global CO2 Emissions (Country-level, Simplified)

### **7. Support & Resources**

*   **Exemplar Notebook:** An annotated example notebook will be provided on Canvas. This model will demonstrate the expected structure for the final submission, including Markdown explanations, commented functions, and presentation of results.
*   **Dedicated Office Hours:** During project weeks, specific TA office hours will be designated as **"Project-Only Help Sessions"** to provide focused support.
*   **In-Class Work Sessions:** Lab sessions during project weeks will be converted into project work sessions.

### **8. Academic Integrity Guidelines**

1.  **Integrity Pledge:** You must include a comment at the top of your final submitted notebook file: `"# [Your Name] certifies that this submission is my own original work in accordance with the university's academic integrity policy."`
2.  **Collaboration Policy:** You may discuss high-level ideas and debugging strategies with peers. However, all code submitted must be your own individual work. Sharing, copying, or viewing another student's project code is a serious violation.
3.  **Use of External Resources:** You may **only** use Python's standard library modules (e.g., `csv`, `math`). The use of external libraries not covered in the course (e.g., `pandas`, `numpy`) is **not permitted**.
4.  **AI Code Generation Tools:** Using AI tools like GitHub Copilot or ChatGPT to generate project code is strictly prohibited. You may use these tools to ask for conceptual explanations or to help debug an error message, but the final implementation must be your own.
5.  **Submission Verification:** You must submit your chosen `.csv` dataset along with your `.ipynb` notebook to ensure reproducibility.
