## Section 3: Requirements Elicitation Techniques

### Learning Objectives
- Identify and distinguish between common requirements elicitation techniques and their specific applications.

### Assessment Questions

**Question 1:** Which requirements elicitation technique is most effective for capturing detailed insights through personal, face-to-face interaction?

  A) Questionnaires
  B) Interviews
  C) Observations
  D) User Stories

**Correct Answer:** B
**Explanation:** Interviews allow for in-depth understanding through direct, flexible dialogue with stakeholders, making them ideal for detailed requirements.

**Question 2:** What is the primary benefit of using workshops during requirements gathering?

  A) Collect large amounts of data quickly
  B) Understand users' tacit needs without discussion
  C) Promote collaboration and shared understanding among stakeholders
  D) Observe user behavior in natural settings

**Correct Answer:** C
**Explanation:** Workshops facilitate collaborative discussion, alignment, and consensus-building among stakeholders, which helps in prioritizing and refining requirements.

**Question 3:** Which elicitation method is particularly useful for understanding implicit or unspoken needs by directly observing users at work?

  A) Questionnaires
  B) User Stories
  C) Observations
  D) Interviews

**Correct Answer:** C
**Explanation:** Observations enable analysts to see real-world workflows and uncover hidden or unarticulated needs that users may not verbally express.

**Question 4:** User stories are primarily used to:

  A) Document detailed technical specifications
  B) Capture high-level, user-centric requirements in simple, manageable statements
  C) Conduct formal requirement reviews
  D) Establish project timelines

**Correct Answer:** B
**Explanation:** User stories are short, simple statements that focus on what the user needs and why, fostering clarity and prioritization in development.

## Section 6: Techniques for Clear and Complete Requirements

### Learning Objectives
- Describe techniques to ensure requirements are clear, complete, and achievable.
- Identify how IEEE standards, SMART criteria, and traceability matrices improve requirements quality.
- Apply SMART criteria to improve ambiguous or vague requirements.
- Develop a basic traceability matrix to connect requirements with design and testing.

### Assessment Questions

**Question 1:** Which technique helps ensure requirements are Specific, Measurable, Achievable, Relevant, and Time-bound?

  A) Traceability Matrices
  B) IEEE Standards
  C) SMART Criteria
  D) UML Modeling

**Correct Answer:** C
**Explanation:** SMART criteria explicitly focus on making requirements well-defined, measurable, attainable, relevant, and time-constrained.

**Question 2:** What is the primary purpose of a traceability matrix in requirements engineering?

  A) To document the project timeline
  B) To ensure every requirement is linked to design, testing, and implementation
  C) To define the scope of the project
  D) To identify stakeholders' roles

**Correct Answer:** B
**Explanation:** A traceability matrix maps requirements to design elements, tests, and tasks, ensuring traceability and completeness.

**Question 3:** According to IEEE standards like IEEE 830, requirements should be written to promote:

  A) Flexibility and ambiguity
  B) Clarity, consistency, and traceability
  C) Lengthy and detailed documentation regardless of relevance
  D) Minimal documentation to save time

**Correct Answer:** B
**Explanation:** IEEE standards emphasize clear, consistent, and traceable documentation to facilitate understanding and validation.

**Question 4:** Which of the following is an example of a SMART requirement?

  A) Improve system performance
  B) Reduce response time to under 2 seconds for data load, within 3 months
  C) Make the interface user-friendly
  D) Enhance security measures

**Correct Answer:** B
**Explanation:** Option B clearly specifies a measurable goal with a deadline, exemplifying SMART criteria.

**Question 5:** What is a potential risk of not using specified techniques like IEEE standards, SMART criteria, or traceability matrices?

  A) Requirements may be too detailed
  B) Project scope may become too small
  C) Requirements may be ambiguous, incomplete, or misaligned with goals
  D) The project team will be overstaffed

**Correct Answer:** C
**Explanation:** Without structured techniques, requirements risk being vague, incomplete, or disconnected from actual project needs.

### Activities
- Evaluate an example requirement such as 'The system shall load data quickly' and revise it to meet SMART criteria, making it: 'The system shall load user data within 2 seconds under typical load conditions during peak hours, to be implemented within 4 weeks.'
- Create a simple traceability matrix for a small feature, linking requirements, design components, and test cases.

### Discussion Questions
- Why is it important to make requirements measurable, and how do techniques like SMART facilitate this?
- How can traceability matrices impact the success of a project? Can you think of a situation where they might prevent issues?
- What challenges might arise when trying to implement IEEE standards in a real-world project, and how could these be addressed?

## Section 8: Common Challenges and Best Practices

### Learning Objectives
- Recognize common challenges such as scope creep, ambiguous requirements, and communication gaps.
- Apply best practices like documentation, stakeholder engagement, and change management to improve requirements engineering.

### Assessment Questions

**Question 1:** Which of the following is a common challenge in requirements engineering?

  A) Clear communication
  B) Scope creep
  C) Well-defined requirements
  D) Stakeholder approval

**Correct Answer:** B
**Explanation:** Scope creep refers to uncontrolled or undocumented expansion of project scope, making it a well-known challenge that can jeopardize project timelines and budgets.

**Question 2:** What technique can help clarify ambiguous requirements?

  A) Ignoring stakeholder input
  B) Building prototypes early
  C) Reducing stakeholder involvement
  D) Avoiding documentation

**Correct Answer:** B
**Explanation:** Prototyping provides visual and functional clarification of requirements, reducing ambiguity and aligning stakeholder expectations.

**Question 3:** Which best practice helps prevent scope creep?

  A) Avoid change requests
  B) Formal change control process
  C) Rely solely on initial requirements
  D) Disregard stakeholder feedback

**Correct Answer:** B
**Explanation:** Implementing a formal change control process ensures that all requested changes are assessed for their impact before approval, helping to contain scope creep.

**Question 4:** Regular stakeholder involvement is important because it:

  A) Increases project delays
  B) Ensures ongoing alignment and reduces misunderstandings
  C) Eliminates the need for documentation
  D) Allows developers to finalize coding early

**Correct Answer:** B
**Explanation:** Frequent engagement with stakeholders improves understanding, clarifies requirements, and minimizes the risk of costly rework due to miscommunication.
